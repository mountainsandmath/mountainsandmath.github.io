Packing 17 Squares in a Square

April 12, 2026

A first look at how square packings behave nicely in some cases and unexpectedly in others.

The Problem

Suppose we want to pack n smaller congruent squares into one larger square region as efficiently as possible. For some values of n, the answer looks exactly how we would expect: orderly, symmetric, and easy to describe. For other values, the geometry becomes much stranger.

Simple Cases: n = 9 and n = 25

When n is a perfect square, things behave very cleanly. If n = 9, we can arrange the squares in a 3 by 3 grid. If n = 25, we can arrange them in a 5 by 5 grid. In general, if n = k², the squares fit perfectly into a square grid with side length k.

These examples are satisfying because the combinatorics and the geometry align perfectly: the number of small squares matches the structure of the larger square.

The Case n = 17

Then we get to 17, and the picture changes completely.

There is no obvious square grid, and the best known constructions look much less orderly. Instead of a neat pattern, the arrangement appears irregular and almost improvised. This is one of the reasons the problem is so visually compelling: even though the pieces themselves are simple, the optimal arrangement is not.

What makes this even more interesting is that the best known construction is tied to an open problem: while there is a most efficient packing currently known, determining whether it is the optimal packing remains unresolved.

I like this example because it shows how a very elementary question can lead to genuinely subtle mathematics. The problem is easy to state, easy to draw, and easy to experiment with, but the underlying optimization problem is much deeper than it first appears.

It is a good reminder that simple mathematical objects often become most interesting exactly when symmetry starts to break down.
